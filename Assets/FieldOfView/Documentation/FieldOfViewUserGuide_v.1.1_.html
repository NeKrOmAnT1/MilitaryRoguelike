OVERVIEW

Field of View is a component that helps to implement visualisation of the area visible by a character. With its help it is possible to implement game mechanics nicely represented by Commandos-like games.

FEATURES

Field of View component:

  • works nicely in 3D mode;
  • can be individually/in bulk configured for any number of characters;
  • is not static, it simulates the behavior of real character who can look around, seek for someone, etc;
  • supports three modes of rotation:
  • ‘calm’ patrol, if no targets have yet appear in the Field of View;
  • ‘peer’ on a target, whenever it is inside the Field of View;
  • ‘cool-down’ after the target has just been gone out of Field of View;
  • supports flexible customisation of rotation patterns;
  • supports different timings for target detection and cool-down periods;
  • has very flexible configuration of interaction with game environment: transparency of every layers can be configured separately;
  • allows to style visible Field of View area with colors or textures;
  • has editor extension that makes the area visible in both: play and edit modes, thus simplifies the process of level design;

therefore Field of View behavior can be simulated in a very realistic way.

USAGE

Single Instance

It is required to execute only two simple steps to add Field of View component to any character:

  1. Select the character that requires Field of View component and create an empty child object. It is important to put the component onto child game object;
  2. Drag-and-drop (in the inspector) the Field of View component  to newly created empty object.

Now, the Field of View is ready to be used. It already has minimum of configuration so it can be tested and tuned right away.

Prefab

To make a prefab out of game character with the Field of View attached it is required to perform one extra step and save the skin material of the component. By default the Field of View component uses dynamically generated materials which are applied to paint its area with color or texture. However this kind of materials cannot be put into a prefab, beforehand they have to be saved into a separate files.

There is a separate settings sections “Material Settings” where exists a button “Persis Material”. After pressing this button “Skin Material Builder” window appears.

It has an explanation whether it is required to save a material or not. Additionally user has to provide “Material Name” and “Material Location” (a full tree of folders related to current project is provided in a dropdown) and save the material.

Now, newly created game object can be converted into a prefab and reused later.

Also, this window can be safely closed (via [x] or [Cancel]), it will lead to the default behavior: attached material will remain dynamic and be serialized within the scene but not to a separate file.

CONFIGURATION

Field of View component has very flexible configuration. The whole set of settings is divided into logical sections. The configuration can be changed in Play mode and the changes will be applied immediately.

General Settings

Set of common characteristics of the component like size and shape.

PROPERTY

DESCRIPTION

Radius

Distance between the center of the Field of View area (owner’s position) and its outer edge.

Angle

Number of degree that represents the width of the Field of View

Resolution

Mesh that builds the Field of View area contains of multiple triangles or sectors. The more sectors are within the Field of View area the better quality of the outer edge it has but the higher computation cost is required.

‘Resolution’ parameter configures number of sectors per degree.

Angle = 45, Resolution = 0.5


Angle = 45, Resolution = 2

Material Settings

Set of material settings. Currently has only one button that allows to persist the material of current FoV holder (or holders in case of multiple selection).

A separate pop-up dialog allows to configure material persistence details. This dialog is invoked by “Persist Material” button.

After saving the material it will be applied to current FoV holder (or holders in case of multiple selection, also in this case the material will be shared between all selected objects).

Visibility Settings

Set of visibility rules and conditions like “when the area must be shown or hidden”.

PROPERTY

DESCRIPTION

Always Visible

Field of View area is always visible. If this option is enabled then conditional visibility rules are disabled and cannot be modified.

Show if Target is spotted/detected

If this option is enabled then the Field of View area will be visible whenever any target exists inside.

Show during Cool Down

If this option is enabled then the Field of View area will be visible during the cool-down time (target has just been gone out of the Field of View area).

Very interesting result can be achieved when both “Show if Target is spotted/detected” and “Show during Cool Down” rules are enabled:

In this case the Field of View area is not visible whenever the target is lost completely. This configuration simulates regular gameplay of stealth genre games.

In case if all the rules are disabled, which means that the area will be always hidden, there is a warning message that explains possible misconfiguration:

Rotation Settings

Set of patterns which are used to specify the way how the area is rotating in time. Each pattern can be defined as a curve by setting particular points. This curve is defined on a grid where horizontal axis represents time and vertical - angle of the deviation (can be positive and negative; 0 is a middle point). There is a possibility to use predefined patterns.

PROPERTY

DESCRIPTION

Default Rotation

Rotation pattern that is used whenever the owner of the Field of View was not yet disturbed by any target, or it happened long enough time ago (after cool-down time interval). Simulates the process of regular patrol of the character.

Cool-down Rotation

Rotation pattern that is used whenever the target has just been gone out of Field of View but cool down period is not yet finished.

At the moment when the target is spotted or detected there is a third rotation pattern being enabled: Look At. It is very simple: middle vector of the Field of View is always pointed to the target (even if the target is moving).

Detection Settings

Set of settings related to target detection with corresponding timeouts.

PROPERTY

DESCRIPTION

Targets Layer

Layer that is checked by the Field of View for the targets to be detected. Targets must have a collider attached.

Detection Time

Maximum time required to check (fill with active color) the area of the Field of View for targets. Detection is initiated whenever the first target appears inside the Field of View.

Cool-down Time

Time interval required to forget the target if it is not in the area anymore (fill with passive color). Cool-down is initiated whenever detected targets have been gone out of the Field of View area. ‘Cool-down Rotation’ pattern is used during this time.

In case if the layer which is used for scanning for a target is not set there is an info message that explains possible misconfiguration:

Transparency Settings

Set of rules that defines a way how the Field of View area interacts with other objects in terms of refraction. Each configuration item consists of the layer name and its transparency value. The latter defines the value in % of the total Field of View radius that remains after passing through any object from particular layer.

COLUMN

DESCRIPTION

Layers

Set of layers with custom transparency level configuration.

Transparency

Transparency value in %.

0% transparency:

75% transparency:

Accuracy Settings

Configuration items related to the component quality characteristics. Some items are performance critical.

PROPERTY

DESCRIPTION

Mesh Accuracy Level

Field of View area is built of a narrow triangles in a way how the umbrella is built: Field of View owner is a center and the circle around (or part of it) is cut into small sectors, which are triangles in fact. Normally each triangle (or sector) of the Field of View has fixed angle based on General Settings but in case if the Field of View collides with an obstacle the mesh itself requires some adjustments to remain smooth. For this purpose an affected sector is divided into subsectors to make outer curve look more natural.

The process of dividing is done in a simple way:

  • Affected sector is divided into two equal sub-sectors.
  • Subsector that does not interfere with an obstacle is ignored, and the one that does - taken for the next iteration.

This simple loop allows to find a new edge that is closer to obstacle, thus the Field of View border will be a bit more smooth. To find a perfect border it can take too many iterations and can lead to performance problems. Current property defines the number of loop iterations.

Recommended value: 5.

Note: subsector borders, start and end, are marked with yellow and green colors respectively.

Mesh Accuracy Level = 2

Mesh Accuracy Level = 5

Mesh Accuracy Level = 10 (perfect match)

In case if the very high level of mesh detalization is selected, which means that there can be unnecessary extra computations, there is a warning message that explains possible misconfiguration:

Debug Settings

Configuration items that are intended to help during the configuration of the whole component.

PROPERTY

DESCRIPTION

Show Sector Edges

Field of View mesh is built of small triangles but their edges are not visible. For debug purpose, for example, whenever it is not clear how the Field of View interacts with other objects it is possible to display edges of each triangle within the Field of View (in editor window but not in game window).

Edges can be of the following colors:

  • Blue: start/end edge of the regular sector
  • Yellow: start of the subsector *
  • Geen: end of the subsector *

* see “Mesh Accuracy Level” configuration.

In case if Mesh Accuracy Level is high then yellow and green edges can overlap with each other.

Skin Settings

Set of configuration items that describes the way how the Field of View area is visualized. Depends on the skin type the view of current settings group is changed.

  1. Color Skin Type

PROPERTY

DESCRIPTION

Skin Type

Selector that allows to choose a way the Field of View area is highlighted. Possible values are:

  • Color, the area is painted with a pair of solid colors;
  • Texture, the area is painted with a pair of user provided textures. Two textures are available out of the box.

Passive Color

This color is used to fill the area of the Field of View whenever there is no targets inside or cool-down time interval has been already passed.

In general it identifies the calm state of the Field of View area.

Active Color

This color is continuously displaces Passive Color starting from the moment when any target appears in the area.

Also it is used to fill the area of the Field of View when the target is recognized (detection time interval has been already passed) and thus is detected.

In general it identifies the alarm state of the Field of View.

  1. Texture Skin Type

PROPERTY

DESCRIPTION

Skin Type

Read the details in a previous section.

Passive Texture

Read the details in a previous section about Passive Color.

The texturing is done in the following way: provided texture is applied on top of the Field of View area in a fixed manner. It means that whenever the Field of View is rotated around its center the texture is not rotated. This allows to observe different parts of the texture depends on rotation angle of the Field of View.

Texturing is done from the center of the image, that is why it is very convenient to specify textures in the form of circle.

Example (look at the image below):

  • At first the Field of View is oriented to the right and at this point the blue sector of the texture is observed;
  • Then the Field of View is rotated to the left, but the texture is not, and from now the orange sector of the texture is observed.

Active Texture

Read the description in the previous section about Active Color.

Life-cycle Callbacks

Feedback from the Field of View can be received via delegates/events at 3 different points in time:

  1. Target Spotted - the target has just come into the Field of View area. Detection time has not yet been passed. Target is marked as spotted.
    Event     TargetSpotted with a single argument GameObject (target) is executed.
  2. Target Detected - the target stayed within the area of the Field of View long enough. Detection time has been already passed. Target is recognized and marked as detected.
    Event     TargetDetected with a single argument GameObject (target) is executed.
  3. Target Lost - the target has just gone out of the Field of View area.
    Event     TargetLost with a single argument GameObject (target) is executed.

Each event is triggered once per object per state change. For example if target A got spotted then “Target Spotted” event will be triggered once and not be triggered until the target A will be lost and spotted again.

Subscription to all events listed above can be done very easy.

First, corresponding functions have to be defined:

private void TargetSpotted(GameObject target) {

    Debug.Log("I guess I spot " + target.name);

}

private void TargetDetected(GameObject target) {

    Debug.Log("Now I can see " + target.name);

}

private void TargetLost(GameObject target) {

    Debug.Log("I cannot see " + target.name + " anymore");

}

And lastly the listener has to be subscribed to the corresponding events:

private void OnEnable() {

    this._fieldOfView.TargetSpotted += TargetSpotted;

    this._fieldOfView.TargetDetected += TargetDetected;

    this._fieldOfView.TargetLost += TargetLost;

}

 

private void OnDisable() {

    this._fieldOfView.TargetSpotted -= TargetSpotted;

    this._fieldOfView.TargetDetected -= TargetDetected;

    this._fieldOfView.TargetLost -= TargetLost;

}

DESCRIPTION

Pauses default (no targets neither spotted nor detected) rotation.

ViewResolution

Get/set the resolution of the component. Allowed range is defined in constants: [MinResolution, MaxResolution].

EnableVisibilityRule(VisibilityRules)

DisableVisibilityRule(VisibilityRules)

IsVisibilityRuleEnabled(VisibilityRules)

Enables/disables/return state of a single visibility rule. Acceptable values are:

  • None
  • AlwaysVisible
  • IfTargetDetected
  • IfCoolDown

Most values are self explanatory, except of “None”. For enable/disable operation it moves all flags to the opposite state from what was requested.

For example the following call:

EnableVisibilityRule(VisibilityRules.None)

will disable all flags.

TargetsLayer

Get/set the value of targets layer of the Field of View.

DefaultRotation

Get/set rotation patterns of the Field of View.

DetectionTime

Get/set time delays of the Field of View.

CurveAccuracy

Get/set the level of mesh detalization of the Field of View. Allowed values range is defined in constants: [MinMeshAccuracy, MaxMeshAccuracy]

This list can be easily extended within the component itself. Existing properties are located inside the “ExternalAPI” code region. Separate code region named “ConfigurationConstants” contains set of fixed configuration arguments (in most cases just range constraints).

Q&A

Q: Field of View does not detect a target, why?
A: There two most popular reasons:
    1. Layer of targets is not configured properly.
    2. Field of View mesh does not overlap with the collider of the target (Y-axis to be checked).

Q: There are multiple Fields of View. Changes applied to the material of one of them are also applied to others, why?
A: Normally material is generated at the moment when the Field of View component is attached to some Game Object thus it is individual and not shared. But it can happen that the Game Object with the Field of View was copied. In this case the material is shared between copies. To solve this problem simply avoid copying. Field of View can be assigned to multiple objects at the same time without any problems. It supports simultaneous configuration of multiple objects.

Q: The Field of View area is partially hidden partially displayed, why?
A: Most probably the Field of View area is covered by any other object. For example if the result looks like this:

it is a result of “z-fighting” (Field of View area and floor surface are covering each other). Changing Y coordinate of the Game Object with Field of View attached will solve the problem.
